# 源码特点

  • 以 Runtime + Compiler 版本的 Vue 为例，它的入口是src/platforms/web/entry-runtime-with-compiler.js。

    • 这个 JS 文件import Vue from './runtime/index'

    • './runtime/index'文件内又import Vue from 'core/index'

    • 'core/index'文件内又import Vue from './instance/index'

    • ./instance/index才是 Vue 真正定义的地方，核心逻辑的开始

    • 这一系列引入文件所做的事情有：拓展$mount 原型方法，植入更新patch方法，config 配置，全局 API 设置。

  • 就是这样一种一层层引入，一层层包装，然后构建出一个完整功能的 Vue，最终通过构建工具打包到 dist 目录。

  • 虽然代码是这样一层层引入构建，从外到内合成的，但是代码的执行逻辑和初始化却是顺着这个引入从内到外的。

# $mount 实现

  • Vue 中是通过$mount 实例方法去挂载 vm 的

  • $mount 方法的入口文件在src/platform/web/entry-runtime-with-compiler.js

  • $mount 方法有二层劫持方法的封装，最内的一层为核心内容，也是层层引入实现

    • 第一层劫持，将 template 模板处理成 render

    • 第二层劫持，处理环境差异（游览器或服务端渲染）

    • 第三层，mountComponent 方法，主体内容如下

      • 触发 beforeMount 钩子

      • 配置更新渲染方法

      • 生成数据监测的观察者

      • 触发``new Vue 的 mounted 钩子

# render 实现

  • 文件定义在src/core/instance/render.js

    • 主要用来把实例渲染成一个虚拟节点

    • 当外部没有设置 render 时，内部会走默认的 render

    • 除了供外部使用的 createElement 方法配置，内部还有一个用于模板编译成的 render 函数使用的 createElement 方法

    • 虚拟节点的文件定义在src/core/vdom/vnode.js，是一个 Class 定义的 VNode 类，内容主要是属性定义

# createElement 实现

  • 这个方法是具体的 VNode 生成。

  • 如果判断是组件，则再走组件创建createComponent。

  • 否则，则直接创建虚拟节点new VNode，涉及文本节点的判断，而且如果存在子级则会递归生成。

# 内部 update 实现

  • 将具体的 VNode 转换为 DOM 并更新到文档树中。

  • 内部涉及到的 patch 方法根据不同环境进行函数柯里化生成的，分支逻辑较多，比较复杂。

  • 更新子级时，会遍历子虚拟节点，递归调用 createElm，运用了深度优先的遍历算法。

  • 实际上整个过程就是递归创建了一个完整的 DOM 树并插入到 Body 上。

# 组件化

# createComponent 实现

  • 同样是生成虚拟节点

  • 在 createElement 内部，里面根据传参的不同，会走 createComponent 组件创建逻辑。

  • createComponent 实现包含三个核心步骤

    • 构造子类构造函数，拿到初始化时合并的一些配置

    • 安装组件钩子函数，内部的 insert 函数负责触发组件级的 mounted 钩子

    • 实例化 VNode，生成虚拟节点

# 组件的 DOM 更新

  • 具体的更新，同样是走内部的 patch 方法，跟 createElement 方法最终走 patch 一样。

  • 在内部运行过程中，组件的更新会去 new 那个通过 createComponent 创建的子类构造函数。

  • 这个更新过程，也是一个递归调用和遍历子树的过程，如果是组件 VNode，则递归这个组件更新流程。

  • beforeMount & mounted

    • 在真正执行$mount 动作的 mountComponent 方法中

      export function mountComponent (vm, el, hydrating ) {
  vm.$el = el

  callHook(vm, 'beforeMount')

  let updateComponent

  // 定义更新界面的方法
  if (process.env.NODE_ENV !== 'production' && config.performance && mark) {
    updateComponent = () => {
      // ...
    }
  } else {
    updateComponent = () => {
      vm._update(vm._render(), hydrating)
    }
  }

  // 监测数据变化
  new Watcher(vm, updateComponent, noop, {// 对象方法 }, true)
  hydrating = false

  // 表明这不是一个组件的初始化过程，而只是new Vue的初始化过程
  if (vm.$vnode == null) {
    vm._isMounted = true
    callHook(vm, 'mounted')
  }
  return vm
}

    • 组件的 mounted 触发

      • 通过 render => createElement 分支逻辑 => createComponent 过程中配置的，mounted 的触发遵循“后进先出”的原理，也就是子级会先从函数栈中弹出，mounted 钩子的触发由子组件依次到父组件。

      • new Vue实例化的 mounted 触发则没这种“后进先出”的特点，是“直线型”过程，钩子函数顺着程序逻辑一步步触发。

      • 路由的<router-view/>走的是new Vue然后初始化其下的组件。

  • beforeUpdate & updated

    • 发生在数据更新的时候，前提是 mounted 已触发过。

    • beforeUpdate 的触发，也是在$mount 动作的 mountComponent 方法中，简化如下：

      export function mountComponent(vm, el, hydrating) {
  // ...

  // 监测数据变化
  new Watcher(
    vm,
    updateComponent,
    noop,
    {
      before() {
        // 是否执行过mounted
        if (vm._isMounted) {
          callHook(vm, "beforeUpdate");
        }
      },
    },
    true
  );

  // ...
}

    • updated 的触发，则是在 flushSchedulerQueue 函数调用的时候，也就是数据监测代码的地方，它的定义在src/core/observer/scheduler.js中

      function flushSchedulerQueue() {
  // ...
  // updatedQueue是更新了的wathcer数组
  callUpdatedHooks(updatedQueue);
}

function callUpdatedHooks(queue) {
  let i = queue.length;
  while (i--) {
    const watcher = queue[i];
    const vm = watcher.vm;
    // 只有满足当前watcher为vm._watcher以及组件已经mounted这两个条件才执行
    if (vm._watcher === watcher && vm._isMounted) {
      callHook(vm, "updated");
    }
  }
}

# 组件注册

  • 全局注册

    • Vue.component 方法内会独立地调用 Vue.extend 生成一个子级构造函数，在 render => createElement => createComponent => update => patch 会被拿到后进行 new 调用。

    • Vue.component 相关代码如下：

      export function initAssetRegisters(Vue) {
  ASSET_TYPES.forEach((type) => {
    Vue[type] = function(id, definition) {
      if (!definition) {
        return this.options[type + "s"][id];
      } else {
        // 省略...
        if (type === "component" && isPlainObject(definition)) {
          definition.name = definition.name || id;
          // 走Vue.extend流程，生成子组件构造函数
          definition = this.options._base.extend(definition);
        }
        // 省略...
        // 注册到this上，也就是Vue函数本身
        this.options[type + "s"][id] = definition;
        return definition;
      }
    };
  });
}

export const ASSET_TYPES = ["component", "directive", "filter"];

    • Vue.extend 过程中，会把 Vue.options 配置合并到子构造函数 options，然后在实例化时，又会执行 merge options 逻辑，把子构造函数的 options.components 合并到 vm.$options.components 上。

  • 局部注册

    • 局部组件也需要生成子级构造函数，不过是在 createComponent 方法内被创建的。

    • createComponent 相关代码如下：

      export function createComponent(
  Ctor: Class<Component> | Function | Object | void,
  data: ?VNodeData,
  context: Component,
  children: ?Array<VNode>,
  tag?: string
): VNode | Array<VNode> | void {
  if (isUndef(Ctor)) {
    return;
  }

  const baseCtor = context.$options._base;
  // 如果是对象（局部组件）
  if (isObject(Ctor)) {
    Ctor = baseCtor.extend(Ctor);
  }
}

    • 局部组件同样有一个合并配置的逻辑，只是跟全局组件相比，后者能混入到所有子级，因为可以被全局访问到，但是局部组件只能局部访问到。

    • 其实，所谓的组件注册，本质就是一个对象混入的过程，全局和局部的注册只是一个混入的层级不同而已。

# 异步组件

  • 同样是走 createComponent 函数，但因为传入的是一个工厂函数，因为会走入异步组件处理逻辑

    export function createComponent(
  Ctor: Class<Component> | Function | Object | void,
  data: ?VNodeData,
  context: Component,
  children: ?Array<VNode>,
  tag?: string
): VNode | Array<VNode> | void {
  if (isUndef(Ctor)) {
    return;
  }

  const baseCtor = context.$options._base;
  // 因为不是对象，不会进入extend构造函数生成逻辑
  if (isObject(Ctor)) {
    Ctor = baseCtor.extend(Ctor);
  }

  // 进入异步组件逻辑
  let asyncFactory;
  if (isUndef(Ctor.cid)) {
    asyncFactory = Ctor;
    // resolveAsyncComponent为关键创建逻辑
    Ctor = resolveAsyncComponent(asyncFactory, baseCtor, context);
    if (Ctor === undefined) {
      // 创建了一个占位的注释VNode，非关键逻辑
      return createAsyncPlaceholder(
        asyncFactory,
        data,
        context,
        children,
        tag
      );
    }
  }
}

  • resolveAsyncComponent 关键代码如下：

    export function resolveAsyncComponent (
  factory: Function,
  baseCtor: Class<Component>,
  context: Component
): Class<Component> | void {
  // resolveAsyncComponent函数会被二次调用，这个时候直接返回
  if (isDef(factory.resolved)) {
    return factory.resolved
  }

  //...

  if (isDef(factory.contexts)) {
    factory.contexts.push(context)
  } else {
    const contexts = factory.contexts = [context]
    let sync = true

    const forceRender = () => {
      for (let i = 0, l = contexts.length; i < l; i++) {
        contexts[i].$forceUpdate()
      }
    }

    // 异步的业务操作后，会被调用
    const resolve = once((res: Object | Class<Component>) => {
      // 生成组件构造函数并保存
      factory.resolved = ensureCtor(res, baseCtor)
      if (!sync) {
        forceRender()
      }
    })

    const reject = once(reason => {//...})

    // 普通函数异步组件处理（如果是Promise异步组件，res就是Promise）
    const res = factory(resolve, reject)

    if (isObject(res)) {
      if (typeof res.then === 'function') {
        // Promise结果的处理
        if (isUndef(factory.resolved)) {
          res.then(resolve, reject)
        }
      } else if (isDef(res.component) && typeof res.component.then === 'function') {
        // 高级异步组件处理
      }
    }

    sync = false
    // return in case resolved synchronously
    return factory.loading
      ? factory.loadingComp
      : factory.resolved
  }
}

  • 这里的重点在于，在 resolve 调用后，forceRender 会被调用，然后调用$forceUpdate，就是调用渲染watcher的update方法，让渲染watcher对应的回调函数执行，触发组件的重新渲染。之所以这么做，是因为Vue通常是数据驱动视图重新渲染，但是在整个异步组件加载过程中是没有数据发生变化的，所以通过执行 $forceUpdate 可以强制组件重新渲染一次。

  • 另外 resolve 执行后，传入的工厂函数 factory 已经缓存了 resolved 子构造函数结果，因此$forceUpdate 调用后，当再次调用 resolveAsyncComponent 时，if (isDef(factory.resolved)) {判断会直接返回结果给 createComponent 函数内的 Ctor。

# 深入响应式原理

# 响应式对象

  • new Vue后，_init 方法会执行到initState函数内

    export function initState(vm: Component) {
  // ...
  const opts = vm.$options;
  initProps(vm, opts.props);
  initData(vm);

}

  • 实际的数据绑定集中在initProps和initData上，它们都利用了Object.defineProperty，同时通过一个内部代理方法，将访问变成了this['属性']的形式。

# 依赖收集

  • 大体来说，就是new Vue后，内部会执行到new Watcher（观察者），将渲染更新方法传入，new Watcher（观察者）内部会调用渲染更新方法，进而触发数据绑定逻辑，数据被访问的过程中，new Watcher（观察者）会被巧妙的加入到与触发数据相关联的订阅器中，之后数据的任何改变都能实现通知变化。

  • 模拟代码

    function observe() {
  let dep = new Dep();
  Object.defineProperty(obj, key, {
    enumerable: true,
    configurable: true,
    get() {
      // 收集依赖 - 关联所有与当前数据属性有关的观察者对象（Watcher）
      dep.depend();
    },
    set() {
      // 派发更新-触发所有与当前数据属性有关的观察者对象（Watcher）进行更新逻辑
      this.notify();
    },
  });
}

function Dep(watcher) {
  this.arr = [];
  // 收集对象
  this.depend = function() {
    this.arr.push(watcher);
    // 将对应的Watcher加入进来
  };
  // 通知对象
  this.notify = function() {
    this.arr.forEach((w) => {
      w.update();
    });
  };
}

function Watcher() {
  // 经过new Vue后，执行到内部逻辑
  // 然后，巧妙地将自身实例挂载到对应数据属性的订阅器（Dep）上
  this.update = function() {
    // 更新界面
  };
}

  • 简单来说，就是observe（数据绑定类）关联订阅器（Dep）,后者关联对应的观察者（Watcher），实现一连串逻辑变化。
